## 开头场景

有一段时间，我判断项目状态主要靠感觉。代码能不能跑，取决于我刚才有没有试过；依赖有没有坏，取决于我什么时候发现；文档和实际功能是否一致，通常要到汇报前才暴露问题。

这种方式在小脚本阶段还能忍，但一旦项目变成前端、后端、模型推理、数据处理一起协作，靠感觉就不够了。最典型的情况是，我改了一个看似无关的配置，前端页面正常，后端接口却报错；或者本地能跑，换一台机器就缺依赖。

第七篇，我给自己的项目补上了一个轻量的 CI/CD 思路和 dashboard。它不是企业级流水线，也不追求复杂部署，而是解决一个研究生个人项目最常见的问题：我能不能快速知道项目现在是否健康。

## 这节做了什么

我先把「项目健康」拆成几个可检查的指标：代码能否安装依赖、测试能否跑通、关键脚本是否还能执行、文档入口是否存在、最近一次构建是否成功。以前这些都靠人工记忆，现在我把它们变成命令和状态。

CI 的部分，我优先做最小检查。比如 Python 项目先检查虚拟环境、依赖文件和核心脚本；前端项目先检查包管理器、构建命令和类型错误；全栈项目再补接口连通性。我的经验是，个人项目的 CI 不需要一开始就覆盖所有场景，但一定要覆盖「最容易在汇报前出事」的路径。

CD 的部分，我没有上来就做复杂自动部署，而是做了可重复的发布步骤。比如打包前清理旧产物，生成版本说明，确认输出目录，最后把可展示的 HTML 或服务地址记录到 dashboard。对个人项目来说，发布的核心不是炫技，而是让自己下次还能复现。

Dashboard 则承担「一眼看懂」的作用。我把知识条目数量、最新批次、平均评分、高价值条目数、LLM 调用次数、Token 用量、估算成本、预算剩余和更新时间放在一个页面里。它不像论文笔记那样长，也不像日志那样碎，而是一个项目入口。每次切换项目时，我先看 dashboard，再决定今天从哪里下手。

## 关键产物

第一个产物是一组标准化检查命令。它们分别负责依赖检查、构建检查、测试检查和关键文件检查。哪怕暂时没有完整测试，也至少能知道项目是不是缺文件、缺依赖、缺入口。

第二个产物是 dashboard 页面。它把原本散落在 README、终端输出和待办列表里的信息聚合到一起。页面不一定复杂，但必须诚实：成功就是成功，失败就是失败，不能为了好看隐藏问题。

第三个产物是发布记录。每次能稳定运行的版本，我都会记录日期、变更内容、验证方式和遗留风险。这样后面出了问题，我不用凭感觉回忆「上次到底改了什么」。

## 我真正学到的

我真正学到的是，项目管理不是等项目大了才需要，而是从第一个可复现版本就应该开始。

以前我会觉得 CI/CD 是企业工程的东西，离个人研究项目很远。后来发现并不是。研究项目也有构建，也有依赖，也有交付对象。导师要看演示，同门要复现实验，自己要在两周后继续接着写，这些都是交付。

Dashboard 改变了我对「进度」的理解。过去我说一个项目完成了 70%，其实很虚。现在我会看：核心流程是否跑通，关键指标是否有结果，展示页面是否可打开，错误是否被记录。进度不再是一种情绪，而是由几个可验证的状态组成。

还有一个很现实的收获：轻量检查能显著减少焦虑。项目越多，脑子里悬着的东西越多。如果每个项目都要靠记忆维护，人会很快疲惫。把状态放到 dashboard 上，相当于把一部分心理负担外置出来。

当然，我也踩过坑。最开始我想把 dashboard 做得很漂亮，结果花了很多时间调整样式，却没有增加多少管理价值。后来我重新收敛：先保证信息准确，再考虑视觉表达。工具页面的第一职责是帮助决策，不是展示设计能力。

## 给后来者的行动清单

1. 为你的项目定义 3 到 5 个健康指标，不要一开始追求全面。

2. 把最常用的启动、测试、构建命令写进 README 或脚本，不要只存在终端历史里。

3. 每个项目保留一个「当前状态」入口，可以是 HTML、Markdown，也可以是表格。

4. 失败状态要直接展示，不要只记录成功案例。问题越早暴露，修复成本越低。

5. 如果没有测试，先写 smoke test。能启动、能访问、能输出结果，就是第一层保障。

6. 发布前固定检查清单：依赖、构建、核心功能、展示入口、遗留问题。

7. 不要为了 CI/CD 引入过重平台。个人项目先用本地脚本和简单工作流就够了。

8. 每次汇报后更新 dashboard。汇报暴露的问题，往往就是下一轮迭代最真实的入口。

## 结尾金句

一个项目是否可靠，不取决于我记得多少细节，而取决于它能不能把自己的状态说清楚。
