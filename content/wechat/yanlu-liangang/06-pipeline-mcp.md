## 开头场景

我最开始做 AI 工具链的时候，状态很像在实验室里临时搭电路：这里一个 Python 脚本，那里一个 Prompt 模板，文件夹里再放几个手工整理的 Markdown。每一步都能跑，但每一步都需要我盯着。

比如整理 AI 技术动态，我要先打开 GitHub Trending 或 RSS，再筛选项目，接着把描述复制给大模型分析，最后手动保存成知识条目。单次看起来不麻烦，可当每天都要采集、分析、归档和展示时，这种碎片化操作会不断吞掉注意力。

第六篇，我做的不是追求一个很炫的智能体，而是把这些散落动作接成一条稳定的 pipeline。我的目标很朴素：少记步骤，少复制粘贴，让工具知道下一步该找谁、该读什么、该产出什么。

## 这节做了什么

这一节我把原本分散的「采集数据、调用模型、整理 JSON、保存文件」整理成一条本地 pipeline。同时，我也预留了 MCP 服务，让外部工具可以查询知识库统计、按 ID 获取文章、搜索已有条目。

这里要区分清楚：pipeline 负责编排流程，MCP 更像是模型和外部工具之间的一层协议桥。MCP 不直接替代 pipeline，它让知识库从“文件夹里的 JSON”变成“可以被工具调用的服务”。

我先做了一件很基础但重要的事：把任务拆成可命名的阶段。以前我会说「帮我整理今天的 AI 项目」，现在我会拆成「采集候选条目」「保存 raw 数据」「调用模型分析」「整理成 articles」「生成 dashboard」。拆开以后，每个阶段的输入、输出、失败原因都更清楚。

接着我为每个阶段定义了产物格式。比如 GitHub 项目必须保留名称、链接、描述、stars、语言和 topics；分析结果必须包含中文摘要、技术亮点、相关度评分、评分理由和标签；整理后的文章必须符合 `knowledge/articles/` 的标准 JSON 格式。这样做的好处是，后续无论是生成 Dashboard、写公众号，还是继续做研究复盘，都不用重新整理一遍。

最后，我把 pipeline 的入口固定下来。不是每次靠临场记忆运行一串命令，而是让同类任务走同一套目录、同一套命名、同一套输出规范。工具链最怕「这次例外一下」，因为例外多了以后，维护成本会回到人身上。

## 关键产物

这一节最重要的产物不是某个脚本，而是一条可重复运行的任务流。它记录了每个阶段的输入、处理方式和输出文件。例如 GitHub 数据进入后，先保存到 `knowledge/raw/`，再分析出摘要和评分，最后生成 `knowledge/articles/` 下的标准知识条目。

第二个产物是统一的 JSON 结构。我给每条知识条目都保留固定字段：标题、来源、source_url、摘要、技术亮点、评分、理由、标签和状态。结构看起来不复杂，但它能把模型输出从「一段好像有用的话」变成「可以继续加工的数据」。

第三个产物是错误记录。以前脚本失败了，我只会重新跑。现在我会记录失败发生在哪个阶段：是采集接口失败、模型输出不是合法 JSON、字段缺失，还是质量检查没有通过。记录之后，下一次优化才有抓手。

## 我真正学到的

我真正学到的是，自动化不是把所有事情都交给 AI，而是把人最容易重复犯错的地方交给流程。

研究生阶段有很多任务并不难，但非常碎。碎任务最消耗人的不是时间，而是切换成本。你刚读完论文，准备写实验计划，突然想起指标还没整理；指标整理到一半，又发现数据集名称没有统一。每一个小断点都会打断思路。

Pipeline 的价值就在这里：它不是替我思考，而是替我守住顺序。该先提取事实，就不要急着写结论；该先保存中间产物，就不要直接复制到最终稿；该记录失败，就不要只凭印象说「好像哪里有问题」。

我也意识到，工程化不是大型公司才需要的东西。一个研究生的小项目同样需要工程化，只是规模更小。目录结构、文件命名、输出模板、错误日志，这些东西都很普通，但它们决定了一个项目能不能连续推进一个月、三个月，甚至半年。

还有一个变化是，我开始更克制地使用大模型。以前我会把一大段内容丢进去，希望它一次性给我答案。现在我更愿意让模型完成清晰的小任务，然后把结果接到下一步。模型能力再强，也需要边界；边界越清楚，产物越稳定。

## 给后来者的行动清单

1. 先不要急着做复杂智能体，先把你每天重复三次以上的动作写下来。

2. 给每个动作命名，例如「论文元信息提取」「实验指标整理」「日报生成」，不要让任务停留在模糊描述里。

3. 为每个阶段规定输入和输出。输入可以是 PDF、代码、日志；输出最好是结构化 Markdown、JSON 或表格。

4. 固定目录结构。原始材料、中间产物、最终文档不要混放，否则后面一定会找不到来源。

5. 先让 pipeline 跑通一条最短路径，再逐步补异常处理。不要一开始就追求完美架构。

6. 每次失败都记录原因。重复出现三次的问题，才值得写成自动修复或校验规则。

7. 模板要短而稳定。栏目太多会增加填写负担，栏目太少又无法沉淀经验。

8. 定期检查产物是否真的被使用。如果生成的文件从来不打开，那它就是流程噪声。

## 结尾金句

真正有用的 AI 工作流，不是让工具显得聪明，而是让人可以稳定地往前走。
