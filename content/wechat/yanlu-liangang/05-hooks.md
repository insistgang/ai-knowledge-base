## 开头场景

我以前很依赖“记得检查”。

写完 JSON，记得看字段有没有缺。跑完采集，记得检查来源链接。让 AI 生成内容，记得判断有没有编造。提交代码前，记得跑测试。

这些提醒听起来很合理，但问题是，人总会忘。尤其是研究生的工作节奏经常被打断：上午改论文，下午开组会，晚上修实验，半夜再回来处理知识库。状态好的时候，我会认真检查；状态差的时候，就容易跳过。

所以我开始想，能不能把一些“应该记得”的事情，变成系统自动执行的检查点。

这就是 Hooks 的意义。

## 这节做了什么

我把知识库项目里的部分质量检查独立成 hooks。

目前仓库里有两个很典型的检查脚本：`hooks/validate_json.py` 和 `hooks/check_quality.py`。前者关注结构，后者关注质量。它们不替我做创造性判断，但能帮我守住底线。

比如知识条目必须是合法 JSON，必填字段不能缺，状态值不能乱写，`source_url` 至少要是基本的 HTTP 链接格式，摘要长度不能无限膨胀，标签不能为空，技术亮点不能只剩空话。这些规则如果靠人工检查，每次都很烦；但写成检查脚本之后，就可以在固定节点自动执行。

这里要说清楚边界：`AGENTS.md` 里确实要求摘要使用中文，但当前 hooks 还没有严格实现“中文摘要检测”。它们已经能检查摘要长度、空洞词、技术信号、字段结构和标签质量，中文表达仍然需要人工或后续规则继续补强。

我对 Hooks 的理解很朴素：它不是为了显得工程化，而是为了减少低级错误进入主流程。

在 AI 知识库这种项目里，低级错误的代价并不低。一个字段名写错，后续 dashboard 可能读不到；一个来源缺失，后续分析就无法追溯；一条编造内容进入知识库，后面生成日报、公众号、研究笔记时都可能被污染。

## 关键产物

这一节的关键产物，是项目里开始出现自动质量守门。

它把一些原本依赖个人注意力的规则，变成可以重复运行的脚本。

我最看重的是两类检查。

第一类是格式检查。JSON 是否能解析，字段是否完整，状态是否符合枚举，路径是否符合约定。这类问题没有讨论空间，应该交给程序处理。

第二类是内容质量检查。摘要是否过长，是否有空洞词，标签是否为空或过宽，技术亮点是否太泛，`source_url` 字段是否存在且格式基本合理。这类检查不一定能判断文章好坏，也不会真的联网验证链接可访问，但能帮助我过滤明显不合格的条目。

这些检查脚本不是孤立存在的。它们会被本地采集脚本和 GitHub Actions 工作流调用：采集完成后先校验 JSON，再做质量评分，再决定是否继续生成 Dashboard 和提交结果。Hooks 的好处是，它让项目质量不再完全依赖“今天我够不够细心”。它把最低标准固定下来，只要流程运行，就会执行同一套规则。

## 我真正学到的

我真正学到的是：工程可靠性不是靠自律堆出来的，而是靠机制托底。

自律当然重要，但自律不稳定。尤其是长期项目里，人会累，会赶时间，会被打断。一个系统如果必须要求维护者每次都高度专注，才能避免低级错误，那它本身就不够稳。

Hooks 让我换了一个角度看质量控制。以前我会问：“我能不能更仔细？”现在我会问：“这个错误能不能不靠我记？”如果能，就把它写成脚本、测试或自动检查。

这对 AI 协作尤其重要。AI 很擅长生成内容，但它不天然保证每次都符合本地规则。我们不能只靠提示词说“请保证字段完整”，还要有外部检查确认它真的完整。提示词负责引导，Hooks 负责验收。

我也意识到，Hooks 不应该一开始就做得很重。项目早期最需要的是抓住高频、明确、可自动判断的问题。比如 JSON 合法性、必填字段、状态枚举、来源链接。这些检查简单，但收益很高。等流程稳定后，再逐步增加更复杂的质量规则。

更重要的是，Hooks 会改变我写代码和写内容的心理状态。因为我知道有一层自动检查在后面接住基础问题，就能把更多精力放在结构设计、判断质量和表达清晰上。

## 给后来者的行动清单

如果你也想给自己的项目加 Hooks，可以从这些问题开始。

1. 列出最近三次最常见的低级错误。
2. 判断哪些错误可以被程序自动发现。
3. 先写最简单的检查脚本，不要一开始追求完整平台。
4. 把必填字段、文件格式、状态枚举作为第一批规则。
5. 让检查脚本能单独运行，方便调试和复用。
6. 在本地脚本、GitHub Actions、生成 Dashboard 等关键节点触发检查。
7. 定期复盘误报和漏报，逐步调整规则。

对研究生来说，Hooks 最大的价值是节省注意力。它不能代替你思考，但能帮你守住那些不该反复犯的错误。

## 结尾金句

成熟的工作流，不是要求自己永远记得检查，而是把该检查的地方变成系统自动停一下。
